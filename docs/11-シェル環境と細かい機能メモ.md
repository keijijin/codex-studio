## 1. Agent Shell とログインプロファイル

### 何ができるようになったか

Agent の **Shell** ツールと **Hooks** の shell アクションは、コマンド実行前に OS ごとのログイン／対話プロファイルを自動で読み込みます。

| 環境 | 読込の例 |
|------|----------|
| macOS / Linux bash | `~/.bash_profile` / `~/.bashrc` / `~/.profile` |
| zsh | `~/.zprofile` / `~/.zshrc`（必要なら bash_profile も） |
| fish | `~/.config/fish/config.fish` |
| Windows PowerShell | `$PROFILE` 相当のユーザプロファイル |
| cmd | `profile.cmd` / `codex-studio-profile.cmd`（あれば） |

組み込みターミナルも同様に、起動直後にプロファイルを静かに読み込みます。

### 運用上の注意

- Agent に `source ~/.bash_profile` を書かせる必要はありません（二重読込やノイズを避けるため）。
- GUI から起動した Electron でも、ターミナルで使う `node` / `pnpm` / `oc` などが PATH に乗りやすくなります。
- 実装: `packages/tools/src/utils/shell-profile.ts`（`wrapShellCommand`）

## 2. 環境変数・`oc login` の引き継ぎ

### 背景

プロファイル読込だけでは足りないケースがあります。

| 状態 | 引き継がれるか |
|------|----------------|
| `oc login` が `~/.kube/config` などに書く認証 | **はい**（次のプロセスからも読める） |
| そのターミナルだけで `export FOO=...` した変数 | **いいえ**（別プロセスには渡らない）→ 下記の同期が必要 |
| アプリ起動元シェルの環境変数 | Electron の `process.env` として一部残る（起動方法依存） |

### 2.1 ターミナルから Agent に同期

1. アプリ下部の **Terminal** で `oc login` や `export` を実行する  
2. パネル右上の **「環境を Agent に同期」** をクリックする  
3. 以降の Agent Shell / Hooks が、そのスナップショットをマージして使う  

- スナップショットは `.codex/.agent-env.captured` に保存（**gitignore 済み**。トークン等が含まれる可能性あり）
- ワークスペースを開き直すとファイルからメモリへ再読込されます
- IPC: `terminal:captureEnv` / `agentEnv:status` / `agentEnv:clear`

### 2.2 恒久設定: `.codex/agent.env`

ワークスペースに次を置くと、毎回 Agent Shell にマージされます。

```bash
# .codex/agent.env
KUBECONFIG=/Users/you/.kube/config
MY_TOOL_HOME=/opt/my-tool
```

- 形式: dotenv 風 `KEY=VALUE`（`export KEY=VALUE` も可、`#` コメント可）
- **秘密情報をコミットしないこと**（必要なら個人用に置き、リポジトリの `.gitignore` に追加）

### 2.3 マージ優先順位

後勝ちです。

1. Electron の `process.env`
2. `.codex/agent.env`
3. `.codex/.agent-env.captured`（またはメモリ上の同期オーバーレイ）

実装: `packages/tools/src/utils/agent-env.ts` / `packages/app/src/main/services/agent-env-service.ts`

### 2.4 OpenShift / Kubernetes の実践例

```bash
# ターミナルで
oc login --server=https://api.example.com:6443 -u myuser
# または
export KUBECONFIG=/path/to/kubeconfig

# → 「環境を Agent に同期」
# → Agent に「oc get pods -n myns」などを依頼
```

`oc login` がデフォルトの kubeconfig を更新しているだけなら、同期なしでもプロファイル＋同一ユーザの HOME で動くことが多いです。  
**セッション限定の `KUBECONFIG` / トークン** を使っているときは同期または `agent.env` を使ってください。

## 3. マルチモデルルーティング（要約）

詳細は [10-マルチモデルルーティングプラン.md](./10-マルチモデルルーティングプラン.md)。

| モード | 挙動 |
|--------|------|
| Fixed | 既定モデル。接続失敗時はフォールバック候補も試す実装あり |
| Fallback | チェーンに沿って 429・接続失敗などで切替 |
| Auto | タスク種別で候補順を決めてから試行 |

- AI パネルの **Fixed / Fallback / Auto** でクイック切替
- 設定画面でフォールバックチェーン編集
- Agent は **ツール実行開始後はモデル切替しない**（副作用回避）

## 4. その他の直近の細かい改善

| 項目 | 内容 |
|------|------|
| E2E 隔離 | `ELECTRON_USER_DATA` を尊重し、スモークテストが実ユーザの設定を汚さない |
| Windows パッケージ | Visual Studio なしでも `node-pty` プリビルドでパッケージ可能 |
| Windows 書込バックアップ | バックアップパスまわりの修正 |
| PowerShell ブートストラップ | ターミナル起動時のプロファイル読込を静かに実行 |
| Electron TLS | システム CA を使った fetch（社内プロキシ等向け） |
| Ollama 可用性 | 接続プローブで未達候補をスキップしやすく |
| Markdown リンク | プレビューからのワークスペース内リンク解決（関連変更がある場合は操作マニュアル参照） |

## 6. セキュリティメモ

- 「環境を Agent に同期」は **現在のターミナル環境全体**（トークン・パスワード相当を含む可能性）を Agent 実行環境に渡します。
- 共有マシンやデモ前には不要な同期をクリアするか、ワークスペースを閉じてオーバーレイを落としてください（`agentEnv:clear`）。
- `.agent-env.captured` はリポジトリに含めないでください（ルート `.gitignore` で除外済み）。
