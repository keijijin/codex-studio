## 1. 目的

Codex Studio を「AI 搭載 IDE」から **自律的な開発エージェント基盤** へ進化させる。  
Claude Code が強い領域（Skills / Hooks / Subagents / Headless）を取り込みつつ、既存の Local-first・承認モデル・Rules を活かす。

### 非目標（当面やらない）

- Cursor 完全互換（Tab 補完・VS Code 拡張エコシステム）を同時追従すること
- Cloud VM / Remote Control（インフラ製品化）を最初から作ること
- Agent Teams のフル協調（Subagents 安定後に検討）

---

## 2. 現状とのギャップ

| Claude Code 機能 | Codex Studio 現状 | ギャップ |
|------------------|-------------------|----------|
| Skills | Rules（`.codex` / `.cursor`）のみ | スラッシュ起動の再利用ワークフローがない |
| CLAUDE.md | Rules 分散 | プロジェクト常駐コンテキストの単一エントリが弱い |
| Permission モード | YOLO の On/Off | ツール種別ごとの allow / ask / deny がない |
| Compact | 簡易履歴 trim | 明示的な要約圧縮がない |
| Hooks | なし | 保存・編集完了などのイベント自動化がない |
| Headless / CLI | Electron UI のみ | CI・無人実行の入口がない |
| Subagents | 単一 Agent ループ | 並列エージェント・結果統合がない |
| Auto Memory | なし | 学習メモリの永続化がない |
| Web 検索 | なし（MCP も未） | 外部情報取得がない |
| Agent Teams / Cloud | なし | 後続フェーズ |

---

## 3. 方針

| 方針 | 内容 |
|------|------|
| **agent-core 共有** | UI / CLI / Subagent が同じ orchestrator・tools を使う |
| **ファイル駆動** | Skills / Hooks / Memory はリポジトリ内の宣言ファイルで管理 |
| **Cursor 互換維持** | `.cursor/rules` は継続読込。CLAUDE.md / Skills を追加読込 |
| **段階リリース** | Phase A → B → C。各 Phase で dogfooding 可能にする |
| **安全既定** | Permission は既定 ask。YOLO は明示オプトインのまま |

---

## 4. フェーズ計画

```
Phase A: 基盤強化（Skills / 常駐 MD / Permission / Compact）   〜 2–3 週
Phase B: 自律運用（Hooks / Headless CLI）                    〜 2–3 週
Phase C: 並列・学習（Subagents / Memory / WebSearch）         〜 3–4 週
Phase D: 拡張（Agent Teams / Cloud / Remote）                 必要時
```

### Phase A — すぐ効く差別化

**ゴール**: `/review` 等の Skills が使え、権限が細かく、長会話が切れにくい

| # | 成果物 | 完了条件 |
|---|--------|----------|
| A1 | Skills ローダー + 実行 | `.codex/skills/`（または互換パス）を読込、チャットで `/name` 起動 |
| A2 | プロジェクト常駐 MD | `CODEX.md` / `CLAUDE.md` / `AGENTS.md` を system に自動注入 |
| A3 | Permission モード | ツールカテゴリごとに `allow` / `ask` / `deny`。設定 UI + 実行時適用 |
| A4 | Compact | 履歴が閾値超過時に要約して継続。手動 Compact コマンド |

### Phase B — 無人・イベント駆動

**ゴール**: 保存時フックと CLI から同じ Agent を回せる

| # | 成果物 | 完了条件 |
|---|--------|----------|
| B1 | Hooks | `onFileSave` / `onAgentComplete` 等でシェルまたは Agent Skill を起動 |
| B2 | Headless CLI | `codex-studio agent "…"`（仮）でワークスペース上の無人実行 |
| B3 | agent-core 分離 | CLI が Electron なしで orchestrator を起動可能 |

### Phase C — 並列エージェント

**ゴール**: 複数サブエージェントを並列実行し結果を統合

| # | 成果物 | 完了条件 |
|---|--------|----------|
| C1 | Subagents | 親 Agent が子タスクを spawn。並列上限・承認集約 |
| C2 | Auto Memory | `MEMORY.md`（または `.codex/memory/`）への自動追記（オプトイン） |
| C3 | WebSearch ツール | 組み込みまたは MCP。許可モデルに従う |

### Phase D — 必要時のみ

- Agent Teams（役割分担・共有ボード）
- Cloud Execution / Remote Control
- チーム共有 Skills

## 5. 提案ディレクトリ構成

```
{workspace}/
  CODEX.md                 # 常駐コンテキスト（CLAUDE.md / AGENTS.md も互換読込）
  .codex/
    rules/                 # 既存
    skills/
      review/
        SKILL.md           # メタ + 手順
      security/
        SKILL.md
    hooks.json             # イベント → アクション
    memory/
      MEMORY.md            # Auto Memory（Phase C）
  .cursor/rules/           # 既存互換
```

## 6. リスクと対策

| リスク | 対策 |
|--------|------|
| Subagents の API コスト爆発 | 並列上限・トークン予算・子 Agent のツール制限 |
| Hooks の無限ループ | フック再入禁止・クールダウン・パス除外 |
| Skills と Rules の重複 | Skills=手順、Rules=制約、常駐 MD=プロジェクト概要と役割分担を文書化 |
| Headless の権限事故 | 既定は ask / deny。CI 用は明示プロファイル |
| 範囲肥大 | Phase A 完了まで Tab 補完・Cloud に手を出さない |

---

## 7. 成功指標

| 指標 | 目標（Phase A 終了時） |
|------|------------------------|
| Skills | 組み込みまたはサンプル 3 本以上が動作 |
| Permission | Write/Shell を deny にすると実行されない |
| Compact | 長いセッションで「履歴上限で突然停止」が減る |
| Dogfooding | Codex Studio 自身の開発で `/review` 相当を常用 |

| 指標 | 目標（Phase B 終了時） |
|------|------------------------|
| Hooks | 保存時に formatter またはテストを起動可能 |
| CLI | CI またはローカルスクリプトから Agent 1 タスク完遂 |
